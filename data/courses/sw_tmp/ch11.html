1. Accessibility (a11y) คืออะไรและทำไมถึงสำคัญ?

คำว่า Accessibility (มักเขียนย่อว่า a11y เพราะมีตัวอักษร 11 ตัวระหว่าง a กับ y) หรือ "การเข้าถึงได้" หมายถึง "การออกแบบและพัฒนาเว็บไซต์ให้ทุกคนสามารถเข้าถึงและใช้งานได้อย่างเท่าเทียมกัน" ไม่ว่าจะเป็นผู้ใช้ปกติ, ผู้ที่มีความบกพร่องทางสายตา (ตาบอดหรือสายตาเลือนราง), ผู้ที่มีปัญหาทางการได้ยิน, หรือผู้ที่มีข้อจำกัดทางการเคลื่อนไหว (ไม่สามารถจับเมาส์ได้)

การทำเว็บให้รองรับ Accessibility ไม่ได้เป็นเพียงแค่ "เรื่องจริยธรรมที่ดีที่ควรทำ" แต่ในหน่วยงานรัฐหรือในหลายประเทศมันคือ "กฎหมาย" (เช่น กฎหมาย ADA ในสหรัฐอเมริกา) นอกจากนี้ เว็บไซต์ที่รองรับ a11y ได้ดียังส่งผลพลอยได้ที่ยิ่งใหญ่คือ ทำให้ SEO (Search Engine Optimization) ดีขึ้นอย่างก้าวกระโดดด้วย เพราะบอทของ Google อ่านและทำความเข้าใจโครงสร้างเว็บได้ง่ายขึ้นมาก

2. โครงสร้างพื้นฐาน: Semantic HTML และ Alt Text

ก้าวแรกที่สำคัญและทรงพลังที่สุดของการทำ a11y คือการกลับไปสู่พื้นฐานอย่างการเขียน Semantic HTML (ตามที่เราเรียนในบทที่ 1) การใช้แท็กให้ถูกความหมาย เช่น <header>, <main>, <nav>, หรือ <button> (แทนการใช้ div มั่วๆ แล้วเอา JavaScript มาผูกคำสั่งคลิก) จะช่วยให้ซอฟต์แวร์ โปรแกรมอ่านหน้าจอ (Screen Reader) เช่น JAWS, NVDA หรือ VoiceOver สามารถเข้าใจและนำทาง (Navigate) ผู้พิการทางสายตาไปยังส่วนต่างๆ ของเว็บได้อย่างถูกต้อง

อีกจุดหนึ่งที่เป็นกฎเหล็กและห้ามลืมเด็ดขาดคือ Alt Text (Alternative Text) ในรูปภาพ ทุกครั้งที่ใช้แท็ก <img> คุณต้องใส่แอตทริบิวต์ alt เสมอ เพื่ออธิบายให้คนตาบอดรู้ว่ารูปนี้คือรูปอะไร (เช่น alt="สุนัขพันธุ์โกลเด้นกำลังวิ่งคาบจานร่อนในสวน")
*ข้อยกเว้น: ถ้ารูปนั้นมีไว้เพื่อตกแต่งสวยงามเฉยๆ ไม่มีสาระสำคัญต่อเนื้อหา ให้ใส่ alt="" (เว้นว่างไว้) โปรแกรมอ่านหน้าจอจะได้รู้ว่าควรข้ามรูปนี้ไป ไม่ต้องอ่านให้รำคาญหู

3. WAI-ARIA (บทบาทสมมติเพื่อช่วยเหลือ)

บางครั้ง HTML ธรรมดาก็ไม่สามารถอธิบายคอมโพเนนต์ที่ซับซ้อนและมีปฏิสัมพันธ์สูง (เช่น แท็บเมนู, อะคอร์เดียน, หรือ Modal Popup) ได้ W3C จึงสร้างมาตรฐานเสริมที่เรียกว่า WAI-ARIA (Web Accessibility Initiative – Accessible Rich Internet Applications) หรือเรียกสั้นๆ ว่า ARIA

  • role="...": ใช้บอกว่า Element นี้รับบทเป็นอะไรบนหน้าจอ เช่น role="alert" หรือ role="dialog"
  • aria-label="...": ใช้กำหนดป้ายชื่อที่มองไม่เห็นบนจอ แต่ Screen Reader จะอ่านให้ฟัง (เหมาะมากกับปุ่มที่มีแต่ไอคอนกราฟิก ไม่มีข้อความ เช่น ปุ่ม 🍔 แฮมเบอร์เกอร์เมนู ควรใส่ aria-label="เปิดเมนูหลัก" ไว้เสมอ)
  • aria-hidden="true": สั่งให้ Screen Reader ข้ามและมองไม่เห็น Element นี้โดยสมบูรณ์ (มักใช้ซ่อนไอคอนตกแต่ง SVG ที่มาจากไลบรารี)

4. การใช้คีย์บอร์ดนำทาง (Keyboard Navigation)

ผู้ใช้บางกลุ่มไม่สามารถใช้เมาส์หรือทัชแพดได้ พวกเขาจะใช้ปุ่ม Tab บนคีย์บอร์ดเพื่อกระโดดไปยังปุ่ม ลิงก์ หรือฟอร์มต่างๆ ตามลำดับ (Focus) สิ่งที่คุณต้องทำในฐานะ Developer คือ ห้ามลบ outline ของ CSS ออกเด็ดขาด! (การเขียน outline: none; ถือเป็นบาปมหันต์ในวงการ a11y เพราะผู้ใช้จะไม่รู้เลยว่าตอนนี้ตัวเองกดเลือกอยู่ที่ปุ่มไหน)

/* วิธีที่ถูกต้อง: อย่าลบของเดิมทิ้ง แต่จงแต่งเส้นขอบโฟกัสให้สวยงามเข้ากับธีมเว็บแทน */
button:focus-visible, a:focus-visible {
  outline: 3px solid #007bff;
  outline-offset: 2px;
}

5. ความเปรียบต่างของสี (Color Contrast Ratio)

สำหรับผู้ที่มีสายตาเลือนราง อาการตาบอดสี หรือแม้แต่คนปกติที่ดูหน้าจอมือถือกลางแดดจ้า หากคุณออกแบบเว็บโดยใช้ "ข้อความสีเทาอ่อน บนพื้นหลังสีขาว" จะทำให้พวกเขาอ่านไม่ออกเลย องค์กร W3C ได้กำหนดมาตรฐาน WCAG ไว้ว่า ความเปรียบต่างระหว่างสีข้อความกับพื้นหลัง สำหรับตัวอักษรขนาดปกติ ต้องมีอัตราส่วน อย่างน้อย 4.5 : 1 (ระดับ AA) คุณสามารถใช้เครื่องมือเสริมในเบราว์เซอร์ DevTools เพื่อตรวจสอบคะแนนสีของเว็บไซต์คุณได้อย่างง่ายดาย
